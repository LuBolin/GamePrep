# Unreal PCG and Procedural Content Workflows

## Cluster scope

**Priority:** P2 for technical artists, tools, world-building and engine generalist roles; P3 for gameplay-only roles unless the project uses procedural worlds.  
**Expected depth:** D3-D4 for workflows and debugging; D5 only for custom PCG element internals and runtime generation source work.  
**Version scope:** UE5.3-UE5.6 preferred. PCG is a plugin-based UE5-era framework; exact node behavior, generation modes, runtime support, API signatures and World Partition integration must be verified in the target branch. [SRC-PCG-001] [SRC-PCG-002] [SRC-PCG-004] [SRC-PCG-005]

PCG is not "randomly scatter meshes." It is a graph-based content pipeline for transforming spatial data, points and attributes into generated content. The official overview describes PCG as a toolset for creating procedural content and tools inside Unreal Engine, ranging from asset utilities to biome generation and entire worlds. [SRC-PCG-001]

The interview signal is data discipline: know what the graph reads, what it writes, how deterministic it is, how generated output is owned/cleaned up, how it works with streaming/cooking/runtime, and how to debug/prove cost.

## Why PCG exists

Hand-authored worlds are flexible but expensive to iterate at scale. Pure code generators can be fast but opaque to artists/designers. PCG sits between those poles:

- designers and technical artists author reusable graph tools;
- programmers can expose parameters, attributes, custom elements and validation;
- points and metadata carry intent through the graph;
- generated output can update interactively in the editor;
- the same system can support local asset utilities, biome tools, set dressing and runtime generation when justified.

PCG does not remove the need for art direction. It changes where rules live. A good PCG tool creates controllable variation, debuggable data and repeatable output. A bad PCG graph creates mystery content that nobody can stabilise, optimise or ship.

## Core mental model

```text
Level / Actor / Volume / spatial input
  -> PCG Component with PCG Graph
  -> nodes generate spatial data and points
  -> attributes, density, tags and metadata guide filtering/transforms
  -> subgraphs / parameters / overrides specialise reuse
  -> spawners or output nodes create assets/actors/instances/data
  -> debug / inspect / regenerate / cook or runtime-generate according to policy
```

Epic's overview says spatial data flows from a PCG Component in the level into the graph, points are generated, filtered and modified through nodes, and resulting points can be used to spawn assets. [SRC-PCG-001]

## Points, density, and attributes

### Points

The overview defines PCG points as locations in 3D space generated by the graph and often used to spawn meshes. Points contain transform, bounds, color, density, steepness and seed information, and can have user-defined attributes. [SRC-PCG-001] [SRC-PCG-007]

Treat a point as a candidate with data, not as a spawned object. A point can be filtered, weighted, tagged, transformed, matched to a data row, turned into an instance, or discarded. This is why PCG workflows benefit from naming attributes clearly: `Biome`, `SlopeClass`, `RoadDistance`, `TreeSpecies`, `bNearWater`, `SpawnWeight`.

### Density

Point density is used by many PCG nodes and is displayed in debug view as a gradient. The overview describes density as the probability of a point existing at a position, where 0 is black and 1 is white in the debug view. [SRC-PCG-001]

Density is not automatically "number of meshes." A later filter, sampler, pruning node, spawner settings or attribute match can change output count. In debugging, compare input point count, density distribution and output point count separately.

### Attributes and metadata domains

PCG attributes are variables stored on graph data. The overview distinguishes static attributes, which start with `$`, from dynamic attributes created at runtime and stored in metadata. It also describes metadata domains:

- `@Data` for single-value data-level attributes;
- `@Points` for per-point metadata;
- `@Elements` for attribute sets. [SRC-PCG-001]

Attribute domain confusion is a common production bug. A graph can "have" the attribute but not on the domain the downstream node reads. Use the attributes list, selector UI and explicit domain prefixes when debugging.

## Graph assets, components, parameters, and instances

### PCG Graph

A PCG Graph asset is the reusable procedural rule set. The official overview compares PCG graph editing to Blueprint or Material graph editing and lists node categories such as Control Flow, Debug, Density, Filter, Metadata, Point Ops, Sampler, Spatial, Spawner and Subgraph. [SRC-PCG-001] [SRC-PCG-003]

Keep graph structure readable:

- use comments and reroutes for intent, not only layout;
- group phases: input, normalisation, candidate generation, filtering, scoring, transform, spawning, validation;
- separate reusable subgraphs from one-off level hacks;
- expose parameters for designer-safe controls;
- leave permanent debug/sanity nodes where they protect important assumptions.

### PCG Component

The overview says the PCG Component holds an instance of a procedural node graph and manages generation in the editor and at runtime. It can be added to an Actor or used through a PCG Volume. [SRC-PCG-001] [SRC-PCG-005]

Architecturally, the component is the bridge between level context and graph execution. It should make ownership and generation policy explicit:

- what inputs are sampled;
- when generation happens;
- whether output is editor-baked, runtime-generated, or both;
- how generated content is cleaned up or regenerated;
- how parameters are overridden for this instance.

### Parameters and graph instances

PCG Graph Parameters are user-created overridable values. The overview compares graph instances to Material Instances and explains that parameters help reuse graphs as instances or subgraphs. [SRC-PCG-001]

Strong PCG tools expose high-level controls such as biome type, density multiplier, seed, slope threshold, asset set, exclusion tag, road offset, LOD/HISM policy or debug mode. Weak tools expose magic scalar values with no units, range or ownership.

## Nodes and data-flow decisions

The node reference is useful not because candidates should memorise every node, but because it shows PCG's categories and failure patterns. The node reference says PCG generates content in editor and runtime by flowing spatial data through graph nodes; points are filtered and modified by listed node categories. [SRC-PCG-003]

Useful node-family mental model:

| Family | Design question | Common failure |
|---|---|---|
| Sampler / Spatial | What geometry/volume/surface/data is being sampled? | wrong coordinate space, unexpected input bounds |
| Density / Filter | Which candidates survive and why? | density treated as hard count; hard constraints only scored |
| Metadata / Attribute | What data travels downstream? | wrong attribute domain/name/type; temporary attributes pollute output |
| Control Flow | Which branch actually executes? | both branches computed unnecessarily or dead branch hidden |
| Subgraph | What is reusable? | parameter names and attribute contracts undocumented |
| Spawner | What is emitted and owned? | too many Actors, wrong instancing policy, missing cleanup |
| Debug | What assumption is being inspected? | debug-only evidence mistaken for shipping behavior |

The node reference explicitly notes useful debug nodes, branch/select behavior, filtering by attribute/tag/type, data counts, attribute deletion, Match And Set, and Static Mesh Spawner-style workflows. It also warns that some debug output does not execute in non-editor/shipping builds and that `Apply On Actor` changes are not revertible by PCG. [SRC-PCG-003]

## Editor generation, runtime generation, and World Partition

Do not treat "PCG can run at runtime" as a design answer. Pick a generation policy:

- **Editor-only/baked:** generation is authoring assistance; output is reviewed and shipped as normal content.
- **Editor-generated with regeneration:** graph remains source of truth during production, but packaged runtime does not regenerate often.
- **Runtime generation:** graph executes in game; must meet frame, streaming, determinism, replication and memory budgets.
- **Partitioned/hierarchical generation:** output and generation cost are managed spatially for large worlds.

Epic's PCG development guide list includes generation modes covering PCG Partitioning, Hierarchical Generation and Runtime Generation. [SRC-PCG-002] [SRC-PCG-004]

The overview also notes World Partition support: PCG assets assigned to a World Partition Data Layer and HLOD Layer generate actors assigned to the same Data Layer and HLOD Layer. [SRC-PCG-001]

Interview answer: editor PCG can be a content tool; runtime PCG is gameplay/runtime system work. Runtime generation needs determinism, streaming policy, collision/navigation implications, save/load rules, multiplayer policy, hitch controls and asset residency planning.

## Determinism, seeds, and reproducibility

Procedural does not mean uncontrolled. At minimum, define:

- graph version;
- seed source and seed override policy;
- input set and order;
- random selection policy and weights;
- platform/build differences that might affect output;
- generated-content ownership and cleanup;
- when output changes should invalidate existing save data or baked content.

If a bug report says "tree blocks mission route sometimes," the team needs a seed, level, graph version, component parameters, input actor set and generated output evidence. Without that, PCG bugs become folklore.

## PCG and gameplay systems

PCG can create content that gameplay systems consume, but do not let generation graphs become unbounded gameplay authority:

- generated cover points should feed AI/navigation validation;
- generated foliage should respect collision, streaming and rendering budgets;
- generated loot containers should be validated by gameplay/item systems;
- generated Smart Objects should still obey query/claim/use/release policy;
- generated spawn points need authority, replay/save and anti-exploit rules in gameplay code.

Use PCG for spatial/data placement and variation. Use gameplay systems for durable state, economy, combat, progression, replication and save/load.

## Debugging workflow

For "the generated forest is wrong":

1. **Scope the failure:** wrong count, wrong location, wrong mesh, wrong transform, wrong layer, wrong runtime/editor state, wrong collision/performance?
2. **Freeze inputs:** level, PCG Component, graph asset/version, graph instance parameters, seed, input actors/volumes/splines/landscape.
3. **Inspect point pipeline:** point count, density, transform, bounds and key attributes after each major phase.
4. **Check domains:** confirm the attribute is on `@Data`, `@Points`, or `@Elements` as expected.
5. **Disable nodes:** use node enable/disable to binary-search the phase that changes output.
6. **Use debug rendering and inspect:** debug render points and inspect attributes on the node that first contradicts expectations. [SRC-PCG-001]
7. **Check spawner settings:** mesh weights, transform/rotation/scale, collision, actor-vs-instance policy.
8. **Check ownership:** generated output cleanup, duplicate generation, World Partition/Data Layer/HLOD assignment.
9. **Check build mode:** editor-only debug nodes, runtime generation settings, cooked asset references and plugin enablement.
10. **Record evidence:** screenshot/debug view, attributes table, seed/parameters and before/after output counts.

Do not begin by tweaking density multipliers. First find the node where expected data becomes wrong.

## Profiling workflow

PCG cost can appear in editor iteration, cooking/build workflows, runtime hitches, rendering, memory, navigation, physics and streaming. Profile by output and phase:

1. Count input data size, generated point count, output Actor/Component/instance count and generated collision/nav impact.
2. Separate graph execution time from spawned content cost.
3. Compare Actors versus instances/HISM/foliage/Nanite where appropriate; do not spawn thousands of rich Actors for static decoration without a reason.
4. Measure runtime generation on target hardware in packaged builds if generation runs during play.
5. Avoid heavy synchronous loads or complex traces in high-frequency runtime generation.
6. Bound generation radii and update frequency; use partitioning/hierarchical modes for large worlds where appropriate.
7. Delete temporary attributes and simplify data before expensive downstream nodes when that is meaningful.
8. Validate streaming, HLOD, Data Layers and memory residency.
9. Capture before/after evidence with the same seed and inputs.

PCG is often blamed for expensive output. Prove whether the graph is expensive or whether it spawned content that is expensive to render, collide, stream, light or save.

## Common bugs and misconceptions

1. **PCG equals random scatter.** It is a data-flow graph; deterministic, parameterised output is often the goal.
2. **Density equals final mesh count.** Filters, pruning, spawners and weights change output.
3. **The attribute exists, so the node should read it.** It may be in the wrong metadata domain or type.
4. **Debug nodes prove shipping behavior.** Some debug output is editor/non-shipping only. [SRC-PCG-003]
5. **Runtime PCG is free because editor PCG felt fast.** Runtime budgets, streaming and hitches are different.
6. **Generated content is not gameplay.** Generated spawn points, cover, loot and Smart Objects need gameplay validation.
7. **Every output should be an Actor.** Static/procedural decoration often needs instances or foliage-like representation.
8. **PCG can mutate external actors safely by default.** Some actor mutation nodes are not revertible by PCG and need care. [SRC-PCG-003]
9. **Changing seeds is enough variation.** Good variation needs constraints, art direction and validation.
10. **World Partition will handle it automatically.** Data Layer/HLOD/streaming assignment and runtime generation policy must be checked.

## Strong interview answer patterns

### "What is PCG in Unreal?"

> PCG is a plugin framework for building procedural content tools through graph-based spatial data flow. A PCG Component feeds level/spatial input into a PCG Graph; the graph generates and transforms points with attributes, density and metadata; spawner/output nodes create content. I think about determinism, parameters, generated-output ownership, debug inspection and whether generation is editor-only or runtime.

### "How do you debug a PCG graph producing wrong output?"

> I freeze the seed, graph version, component parameters and input actors, then inspect the point pipeline phase by phase. I compare point count, density, transform and key attributes after sampling, filtering, metadata operations and spawning. I check attribute domains and types, disable nodes to isolate the first bad phase, inspect the attributes list/debug render, and only then change graph rules.

### "How do you make PCG production-safe?"

> I expose named parameters with ranges and units, define seed/version/input contracts, separate authoring generation from runtime generation, keep output ownership/cleanup explicit, validate generated gameplay hooks through gameplay systems, profile graph execution separately from spawned content cost, and verify cook/World Partition/Data Layer/HLOD behavior in target builds.

### "When should PCG run at runtime?"

> Only when runtime variation is a product requirement and the graph can meet target budgets. Otherwise editor-baked or authoring-time PCG may be simpler and safer. Runtime PCG needs deterministic inputs, bounded frequency/radius, asset residency, streaming policy, save/load and multiplayer authority rules, plus hitch/memory profiling on target hardware.

## What a three-year engineer should know

Know PCG's graph/component/point/attribute model; density and metadata domain basics; graph parameters/instances; node family roles; editor versus runtime generation trade-offs; World Partition/HLOD/Data Layer awareness; debugging with node debug/disable/inspect and attributes list; and profiling graph execution versus output cost.

**Specialist depth:** custom C++ PCG elements/settings, data types internals, runtime partition generation, external data interop, PCG/World Partition source behavior, deterministic large-world generation, multiplayer procedural authority, editor tool packaging and generated-content version migration.

## Hands-on verification task

Build the Project 6 PCG biome tool extension:

1. Create a PCG Graph that samples a volume/landscape/spline and generates points.
2. Add slope/water/road-distance-style attributes and density shaping.
3. Filter points by hard constraints and separate preference scoring from validation.
4. Use Match And Set or equivalent data-driven selection for mesh/type variation.
5. Expose graph parameters for density, seed, asset set and debug mode.
6. Create a graph instance with different parameters.
7. Add a permanent debug/sanity checkpoint for a key assumption.
8. Spawn output using an instance-friendly representation where possible.
9. Verify generated content cleanup/regeneration.
10. Capture 1x/10x/100x output cost and compare graph time versus render/collision/memory impact.
